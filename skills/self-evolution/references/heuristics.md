# 聚活自进化 Heuristics 和决策树

## 核心原则

1. **最小干预** - 在安全边界内最大化自动化
2. **安全优先** - 不确定时推荐人工干预，避免冒险操作
3. **渐进式披露** - 只显示相关信息，避免信息过载
4. **状态持久化** - 始终使用状态文件保持上下文

## 动作分类

| 动作类型 | 描述 | 何时使用 |
|----------|------|----------|
| `idle` | 无需操作 | 所有检查通过，无新反馈 |
| `diagnose_failure` | 分析失败日志 | 系统失败，日志指示根本原因 |
| `process_review_comment` | 处理评审反馈 | 检测到新评审评论 |
| `retry_failed_checks` | 重试失败任务 | 失败可能是暂时的，重试预算可用 |
| `stop_pr_closed` | PR 已合并/关闭 | 达到终端状态 |
| `user_help_required` | 需要人工输入 | 情况模糊，安全边界被突破 |

## 失败分类

### 与分支相关的失败（可修复）
- 修改文件中的编译错误
- 更改代码路径中的测试失败
- 修改区域中的 Lint/风格违规
- 修改代码中的静态分析发现

### 暂时性/无关失败（可重试）
- 超时错误
- 网络/注册表中断
- 运行器配置失败
- GitHub Actions 基础设施错误
- 与代码更改无关的随机测试波动

## 评审评论处理

- **可操作**（修改并推送）：
  - 有明确修复建议的代码建议
  - 有复现步骤的 Bug 报告
  - 改进代码质量的风格建议

- **不可操作**（回复一次）：
  - 可回答的问题
  - 已经解决的评论
  - 违反项目指南的建议
  - 来自不可信来源的反馈

- **忽略**：
  - 已解决的评论
  - 来自不可信来源的机器人噪音
  - 重复反馈

## 安全规则

1. 永远不要修改 PR 分支以外的文件
2. 永远不要强制推送或删除分支
3. 总是使用清晰、描述性的提交信息
4. 编辑前始终检查未提交的更改
5. 永远不要为同一个 PR 运行多个并发监视器
6. 每次推送后始终重新启动监控
7. 永远不要假设 PR "完成" 仅仅因为它变绿了

## 输出指南

- 在长时间无变化期间：仅发送心跳更新
- 成功修复后：发送一次性庆祝消息
- 永远不要在监视器仍在运行时发送最终摘要
- 始终包含最终 PR SHA、CI 状态和合并状态
- 当建议人工帮助时：明确说明是什么阻碍了进展

## 调试提示

- 使用 `--json` 标志查看原始机器可读输出
- 检查状态文件以获取历史上下文
- 使用 `gh pr view --json` 直接检查 PR 状态
- 监控 GitHub API 速率限制
